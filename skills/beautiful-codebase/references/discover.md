# Discover · Phase 1 探测 / 索引 / 清单生成

> **何时读**：Phase 1 Discover 开始时；以及 Phase 0 决定是否提示 `codegraph init` 时。

整个 Phase 1 的目的：把目标项目从"一坨源码"变成"一份可信清单 + 一组按需读的证据底座"。
做完之后，主 Agent **不需要再读一行业务代码**，就能完成 Plan / Build / Review。

工件清单（所有路径都在 `<project>-analysis/discovery/` 下）：

| 文件 | 写者 | 形态 | Phase 2/4 谁会读 |
|---|---|---|---|
| `tier.json` | `tier-select.sh` | JSON 单文档 | 全 Phase 1/4 |
| `inventory.json` | `inventory.sh` | JSON 单文档 | Phase 2 Plan + Phase 5 Coverage Audit |
| `size-tier.json` | `buckets.sh`（写入） | JSON 单文档 | Phase 2 Plan |
| `buckets/_summary.json` | `buckets.sh` | JSON 单文档 | Phase 2 Plan |
| `buckets/bucket-NN-<slug>.json` | `buckets.sh` | JSON 一文件一桶 | Phase 4 每节 Step A SubAgent |
| `business-evidence/comments.jsonl` | `business-evidence.sh` | JSONL | Phase 4 每节 Step A.5 |
| `business-evidence/tests.jsonl` | 同上 | JSONL | 同上 |
| `business-evidence/schema.md` | 同上 | Markdown | 同上 |

## 1 · 探测工具与 tier 选择

由 Phase 0 已经跑过的 `scripts/probe-tools.sh --target <path> --json` 提供。它检测：

- `codegraph` 是否在 PATH（及版本），目标项目是否已 `codegraph init`（`.codegraph/`
  存在）。
- `rg` 是否在 PATH（ripgrep）。
- `grep`（POSIX）。

输出 tier 标签：`codegraph-indexed` / `codegraph-installed` / `rg` / `grep` / `none`。
后续 Phase 1 子流程通过 `lib/query.sh` 调度——它把"列文件 / 查符号 / 文本搜索 / 探索"
四类原语映射到当前 tier，**每个 Discover 脚本都用 lib/query.sh，没有硬编码工具调用**。

`tier-select.sh` 接收 probe JSON + 用户在 Phase 0 的选择（仅当 tier 是
`codegraph-installed`），写出最终的 `tier.json`。如果 codegraph 已装但项目未索引，
**必须先停下让用户选**（见 §2），**绝不静默执行 `codegraph init`**——那会向目标项目
写 `.codegraph/`，属于 medium-risk 写。

## 2 · 用户决策点 · 是否运行 `codegraph init`

仅当 probe tier = `codegraph-installed` 时触发。Phase 0 主 Agent 必须给出：

```
目标项目没有 codegraph 索引（.codegraph/ 不存在）。要让分析最准确，建议现在跑一次
`codegraph init`（约 30 秒到几分钟，会向项目根写一个 .codegraph/ 目录）。

请选：
  A · 现在跑 codegraph init（推荐 · 精度最高）
  B · 不用，降级到 rg / grep（更快，符号查询精度略降）
  C · 我自己稍后跑，到时再回来；现在停下
```

对应 `tier-select.sh --choice {init|downgrade|stop}`：

- `init`：脚本会先回声提示，然后调用 `codegraph init <target>`。失败则非零退出。
- `downgrade`：rg 可用则用 rg；否则 grep；都没有则报错。
- `stop`：退出码 2，主 Agent 停止 Skill，用户自己跑完后重启。

## 3 · 生成 inventory.json

由 `inventory.sh` 产出。**这是真值集**——后续所有 Coverage Audit 都对照它来做。

字段：

```json
{
  "target":   "<absolute path>",
  "snapshot": "<ISO timestamp>",
  "tier":     "<from tier.json>",
  "files": [
    {"path":"src/foo.py", "language":"python", "bytes":1234,
     "loc":56, "sha":"<sha1>", "excluded_reason": null},
    {"path":"vendor/lib.js", "excluded_reason":"vendored"}
  ],
  "stats": {"totalFiles":..., "analyzedFiles":..., "excludedFiles":...,
            "totalLoc":..., "byLanguage":{"python":180, ...}}
}
```

**排除规则**（每条对应一个 `excluded_reason` 取值）：

| Reason | 触发路径模式 |
|---|---|
| `vendored` | `vendor/`, `node_modules/`, `vendored/`, `third_party/`, `third-party/` |
| `generated` | `dist/`, `build/`, `out/`, `target/`, `.next/`, `.nuxt/`, `coverage/`, `*.min.{js,css,map}`, `*.generated.*`, lockfiles, `*_pb.go` / `*_pb2.py`, 首行含 `@generated` / `Code generated by` / `DO NOT EDIT` |
| `fixtures` | `fixtures/`, `testdata/`, `__fixtures__/`, `__snapshots__/` |
| **静默跳过**（不进 inventory） | `.git/`, `.constella/`, `.claude/`, `node_modules/.cache/` |

排除项**仍然出现在 inventory.files 列表里**（只带 `path` + `excluded_reason`），这样
Phase 5 Coverage Audit 能区分"已知排除"和"漏分析"。静默跳过则**不出现**——它们不属
于分析域。

**性能**：脚本采用流式 JSON 拼接（开 `[` → 逐行 append → 闭 `]`），内存占用与文件数
无关；对 10k+ 文件项目也稳定。

## 4 · 生成 buckets/

由 `buckets.sh` 产出。详细策略见 `references/bucket-strategy.md`。简言之：

- 读 `inventory.json` 的 `stats.analyzedFiles` 决定 size tier（`<100` / `100-1k` /
  `1k-10k` / `>10k`）。
- 按 tier 选 bucket 策略：单桶 / 目录 roll-up / codegraph 模块 / 两层目录。
- 每桶目标 ~5k LOC；超出 1.5× 则按顺序切片。
- 桶字段：`id`, `scope`, `rationale`, `files[]`, `loc`, `language[]`, `isEntryHeavy`,
  `evidenceStrategy`。
- 入口启发：含 `controller`/`handler`/`router`/`webhook`/`main.*` 等的桶被标
  `isEntryHeavy: true`，提示 Step A SubAgent 升级到 entry-files-verbatim 证据。

附产物：

- `buckets/_summary.json`：bucket id 列表 + tier + 策略 + evidenceStrategy。
- `size-tier.json`：单文件方便 `codebase-brief.sh` / Phase 2 Plan 一行读取。

## 5 · 生成 business-evidence/

由 `business-evidence.sh` 产出。详细规则见 `references/business-evidence-collection.md`。
6 个文件覆盖 6 类证据来源：

| 文件 | 内容 | 谁用 |
|---|---|---|
| `comments.jsonl` | 跨语言文档注释（JSDoc / Javadoc / docstring / Rust doc / Go `//`） | Step A.5 Business SubAgent |
| `tests.jsonl` | 测试函数名 + 框架（pytest / Go test / JUnit / Jest / it/describe） | 同上 |
| `schema.md` | DDL / 迁移 / ORM struct tag / Prisma 等的简介 | 同上 |
| `configs.md` | 顶层 config 文件 + enum / const 候选 | 同上 |
| `docs.md` | README / `docs/` / 顶层 `*.md` 第一非空行 | 同上 |
| `commit-themes.md` | 近 200 commit message 按 conventional-commit prefix 聚类 | 同上 |

**所有提取都是 best-effort**——不是 ground truth。Step A.5 SubAgent 引用时必须带
`[证据: file:line]`，让 Reviewer 可以回溯到这里来。

## 6 · 生成 codebase-brief.md

由 `codebase-brief.sh` 产出。结构：

1. **At a glance**：文件 / LOC / size tier / tool tier / >10k honesty rule（如果触发）。
2. **Language mix**：按文件数排序的语言分布表。
3. **Top buckets**：scope + LOC + 是否 entry-heavy 标记。
4. **Entry-point sniff (preview)**：9 类角色的粗糙正则计数；只是"feel"，Section 05
   会重新精扫。
5. **Business evidence summary**：comments / tests / schema / configs / docs / commit
   主题的 count + 标签。
6. **Auto-decisions for Phase 2 Plan**：推荐 reader profile + 理由、可选 Section 列表、
   是否允许声称 100% coverage。
7. **How to use this brief**：明确"主 Agent 从此不再读源码"的指令。

这是 Phase 2 主 Agent 写 plan.md 前**唯一必读**的项目级文档。

## 7 · 写出 tier.json + size-tier.json

`tier.json` 由 `tier-select.sh` 写，记录最终选定的 tool tier + tool 版本 + 决策来源
（`auto` / `user-init` / `user-downgrade`）+ 注释列表。

`size-tier.json` 由 `buckets.sh` 写，记录文件数自动分档（`<100` / `100-1k` /
`1k-10k` / `>10k`）+ bucket 策略 + 证据策略。

**`>10k` honesty rule**：当 `size-tier.json.sizeTier == ">10k"` 时，主 Agent 必须在
进 Phase 2 前显式提示：

```
本项目规模 > 10k 文件，强制推荐 reader profile 降级到 archaeology · 70%；
Coverage Annex 强制开启；本报告不会声称 "100% coverage"。
```

用户在 Checkpoint 1 仍可选回 architecture-review / onboarding，但 70% 保留比例与
"无 100% claim" 是不可突破的硬规则。

## 8 · Phase 1 完成自检（主 Agent 内联，无 SubAgent，无文件）

写完上述工件后，主 Agent **就地**核对 5 条：

1. **tools.json 存在且 tier 已写出**——`tier.json` 中 `tier` 不是 `none`。
2. **inventory.json 文件计数与目录大小相符**——`stats.totalFiles` 与 `git ls-files`
   或人眼粗估一致；`analyzedFiles + excludedFiles ≤ totalFiles`。
3. **buckets/_summary.json 与 inventory 一致**——bucket 文件路径集合是 inventory
   `analyzed` 文件集合的子集（split + sort -u 应该等同）。
4. **business-evidence/ 六件齐**——comments.jsonl / tests.jsonl / schema.md /
   configs.md / docs.md / commit-themes.md 都落盘了，即使内容是"(no DB schema
   found)" 也算合规。
5. **codebase-brief.md 读得通**——主 Agent 自己快速通读一遍：语言占比、top buckets、
   entry-point sniff、business evidence summary 这四块都不是空白；推荐的 profile 与
   项目规模一致。

任一条 fail → 重跑对应脚本；都通过 → 进 Phase 2 Plan。**这一步绝不开 SubAgent**：
内容量小、文件刚写完上下文热，开 SubAgent 反而慢。

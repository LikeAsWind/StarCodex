# beautiful-codebase v0.1.0 → v0.2 follow-ups

This document captures every non-showstopper finding from the v0.1.0 dogfood run
on StarCodex (see `.constella/tasks/06-21-dogfood-beautiful-codebase-on-starcodex-phase-h-i/`).
v0.1.0 ships; these items are scheduled for v0.2.

**Total findings: 16** (1 high carry-over + 15 items, ranging high/med/low).

Themes:
- **Reacticle 0.2.6 API drift** (F2, F6, F10, F11) — the skill's prompts and references
  assume reacticle exports that don't exist in the published 0.2.6 package
  (`terminal` theme, `<Code>` / `<Callout>` / `<SourcePointers>` / `<Mermaid>`
  components, `<CodeBlock children>` / `<Table header>` shapes). Workarounds were
  applied during dogfood; the skill should align with the published reacticle API.
- **Skill script defects** (F12, F13, F14, F15) — `density.sh`, `claim-trace.sh`,
  `Article.tsx` footer auto-inject, and `html-to-pdf.sh` Windows path handling all
  have regex / wiring / portability bugs.
- **Theme integration** (F2 → terminal blocked on reacticle update; v0.2 priority)
- **Pre-flight ergonomics** (F1, F9) — `.gitignore` and scaffold restore protocol.
- **Detection / heuristic precision** (F3, F4) — `codegraph files --json` and
  `business-evidence.sh` doc-shape filter.
- **Verification harness limits** (F5, F8) — hard-checkpoint interactivity and
  anti-hallucination physical-isolation cannot be SubAgent-validated; both
  require real interactive user invocation.
- **Spec / batch-prompt alignment** (F7) — filename convention drift between
  `section-build.md` §1.1 and downstream task batch prompts.

### F2 · `terminal` theme not in reacticle 0.2.6 → `tsc` fails

**Severity**: high
**Area**: scaffold + reacticle ThemeProvider enum
**Discovered**: dogfood batch A+B
**Reproduction**: `bash scripts/scaffold.sh ./out --theme=terminal` writes
`<ThemeProvider theme="terminal">` into `article/main.tsx`. The scaffold's own
`tsc --noEmit` emits: `TS2322: Type '"terminal"' is not assignable to type
'"tufte" | "press" | "shannon" | "vignelli" | "knuth" | "freddie" | "andy" |
"bodoni" | "bayer" | "fuller" | "sottsass" | undefined'`. The `terminal` theme
authored as `theme-profiles/terminal.md` (276 lines) is a write-only AI authoring
profile; reacticle 0.2.6 published to npm does not include `terminal` in its
runtime theme union.
**Suggested fix**: Either (a) fork reacticle and publish a 0.3.x build that
includes the `terminal` theme tokens + ThemeProvider enum entry; (b) ship a
local CSS-vars override sheet from the skill that bypasses reacticle's enum
(work around with `theme={"terminal" as any}` + injected `<style>`); or (c)
downgrade scaffold default + Plan Checkpoint 1 recommendation to `tufte` for
code-analysis reports until terminal lands (this is what v0.1.0 already does
per PRD Q5). The theme-profiles/terminal.md design document stays in place as
the spec for the v0.2 work.

### F3 · `codegraph files --json` unsupported in codegraph 1.0.1

**Severity**: low
**Area**: `scripts/lib/query.sh`
**Discovered**: dogfood batch A+B
**Reproduction**: `inventory.sh` prints "codegraph files --json unsupported on
this version, falling back to git ls-files". Means `lib/query.sh`'s
`bc_query_files` attempts `codegraph files --json` which doesn't exist in
codegraph CLI 1.0.1. Silent fallback to `git ls-files` worked, so functionally
fine.
**Suggested fix**: Check `codegraph --help` for the actual file-enumeration
subcommand in 1.0.1 (likely `codegraph ls` or the `codegraph query files-by-pattern`
form). Use it when tier is `codegraph-indexed`; or delete the dead `codegraph
files --json` path. The skill's tier-aware abstraction is currently bypassing
codegraph for file enumeration on the only codegraph version that exists.

### F4 · `business-evidence.sh docs.md` scans every inventory file (incl. binaries)

**Severity**: low
**Area**: `scripts/discover/business-evidence.sh`
**Discovered**: dogfood batch A+B
**Reproduction**: `docs.md` output contains:
```
- `.gitignore` — ### IntelliJ IDEA ###
- `LICENSE` —                                 Apache License
- `README.md` — <p align="center">
- `assets/banner.png` — ▒PNG
```
`.gitignore` / `LICENSE` are non-business noise; `banner.png` line is binary
garbage (literal `<89>PNG` magic bytes). On a real project this could surface raw
binary into the AI's reading material.
**Suggested fix**: Per `references/business-evidence-collection.md` §6, filter to
`*.md` / `README*` / `LICENSE*` / `CHANGELOG*` / `CONTRIBUTING*` / `docs/**.md`.
The spec already lists the include rules; the script just doesn't enforce them.

### F5 · AC-H4 hard-checkpoint halt cannot be SubAgent-validated

**Severity**: med
**Area**: dogfood verification harness (not skill bug per se)
**Discovered**: dogfood batch C+D
**Reproduction**: Parent task AC12 / this task AC-H4 requires "3 hard
checkpoints halt independently and ask separately". In the dogfood, all
Checkpoint 1 (5 decisions), 2 (2 decisions), and 3 (1 decision) were
pre-recorded per PRD R2.3 / R2.4 / R2.5. The main agent did not literally
stop and prompt the user 8 times — this was the PRD-accepted compromise to
let a SubAgent-driven dogfood complete end-to-end.
**Suggested fix**: Validate the hard-checkpoint behavior through a real
interactive user invocation: human types `/beautiful-codebase`, observes
whether the skill stops 5 times in Checkpoint 1, 2 times in Checkpoint 2, and
1 time in Checkpoint 3, with each decision item asked as an independent
question (not packaged into a yes/no). This validation cannot be done by
SubAgent.

### F6 · Reacticle 0.2.6 doesn't export `Code` / `Callout` / `SourcePointers` / `Mermaid`

**Severity**: high
**Area**: `references/component-policy.md` + `references/source-pointers.md` +
reacticle 0.2.6 published API
**Discovered**: dogfood batch C+D
**Reproduction**: `component-policy.md` §2 lists `Code`, `Callout`, `Mermaid`,
`SourcePointers` as "唯一可用 reacticle 组件" to be imported. `prompts/step-b-writing.md`
example imports all of them. `source-pointers.md` defines a `<SourcePointers
pointers={...}>` API. None of them exist in the published reacticle 0.2.6 package
— only `Aside`, `Detail`, `CodeBlock`, `Table`, `Hero`, `Lead`, `Article`,
`ThemeProvider`, etc. are exported. Step B SubAgents that follow the prompt
verbatim will all fail `tsc`. v0.1.0 dogfood worked around with:
- `<code>` (native HTML) for `<Code inline>`
- `<Aside tone="principle">` for `<Callout tone="info">`
- `<Detail summary="..."> + <ul>` for `<SourcePointers>`
- `<pre className="mermaid">` + a top-level `mermaid.initialize({ startOnLoad:
  true, theme: "neutral" })` import for `<Mermaid>`
**Suggested fix**: Either (a) publish a reacticle 0.3.x that exports these
components, then update the skill's prompts/refs to use them; or (b) rewrite
`component-policy.md` + `source-pointers.md` + `prompts/step-b-writing.md` to
use only what reacticle 0.2.6 actually exports. Option (b) is faster and
unblocks v0.2; option (a) is the long-term clean design.

### F7 · evidence/business filename convention drift

**Severity**: low
**Area**: `references/section-build.md` §1.1 vs task batch prompts
**Discovered**: dogfood batch C+D
**Reproduction**: `section-build.md` §1.1 specifies `sections/NN-evidence.md`
and `sections/NN-business.md` (no slug — just numeric prefix). The Phase D / E
dogfood task batch prompts both write `sections/NN-<slug>-evidence.md` and
`sections/NN-<slug>-business.md` (with slug). The dogfood followed the batch
prompt convention because it was the active instruction.
**Suggested fix**: Pick one convention and propagate. The slug-bearing form
is more human-readable when grepping; the slug-less form keeps file pairs
short and aligns with the .tsx-vs-md hint that .tsx is human-facing while
evidence/business are tool-facing. v0.2 should pick one and update both
`section-build.md` and the implicit task-batch convention.

### F8 · "physical isolation" overstated when host can't filter SubAgent tools

**Severity**: low
**Area**: `prompts/step-b-writing.md` permissions line + SKILL.md 反幻觉 wording
**Discovered**: dogfood batch C+D
**Reproduction**: `prompts/step-b-writing.md` says `permissions: 仅允许 Read 上述
输入文件;禁止透传 Glob / Grep / Read 源码权限`. SKILL.md calls this "反幻觉物理
防线" (physical anti-hallucination defense). In Claude Code, SubAgents inherit
the parent's tools by default; there's no per-SubAgent tool filtering yet. So
the prompt's `permissions:` line is a contract (which the SubAgent should
honor) but not a physical isolation (which the host cannot enforce). The
skill's wording "物理防线" overstates the guarantee — it's actually contract +
self-discipline + post-hoc audit.
**Suggested fix**: Soften wording to "合约级 + 物理级（在 host 支持时）反幻觉
防线"; or call out explicitly that when `permissions:` cannot be enforced by
the host, the Section Reviewer's claim-trace audit becomes the actual
guarantee, and the prompt's `permissions:` line is the upstream best-effort.

### F9 · scaffold overwrites existing `discovery/*.json` stubs when re-scaffolding

**Severity**: low
**Area**: `scripts/scaffold.sh` + dogfood restore protocol
**Discovered**: dogfood batch E
**Reproduction**: Dogfood batch C+D did `mv discovery /tmp/backup; rm -rf
./workspace; scaffold; mv /tmp/backup ./workspace/discovery`. The scaffold
generated stub `discovery/*.json` files (empty `{}`); the `mv backup` moved the
real data **into a nested `discovery/bc-discovery-backup/`** subfolder rather
than restoring at top level (because `discovery/` now existed). Took batch E
Step 0 to detect and `cp` the real data back to canonical paths.
**Suggested fix**: Either (a) the scaffold detects an existing `discovery/`
folder and refuses to overwrite (asks user "merge / overwrite / abort"); or
(b) document the rename-then-scaffold-then-rsync workflow in `references/scaffold.md`
explicitly. (a) is more robust; (b) is faster.

### F11 · `<CodeBlock>` and `<Table>` API drift in reacticle 0.2.6

**Severity**: low
**Area**: `prompts/step-b-writing.md` examples ↔ reacticle 0.2.6 API
**Discovered**: dogfood batch E
**Reproduction**: Reacticle 0.2.6:
- `<CodeBlock>` uses `code={...}` prop, NOT `children`. Prompt example showed
  `<CodeBlock>{`...`}</CodeBlock>` → TS2322.
- `<Table columns={...}>`: real `columns[].label` (NOT `header`); real `columns[].align:
  "left" | "center" | "right"` (NOT `start` / `end`). Prompt's implicit example
  with `header` + `start`/`end` → TS2322.
Dogfood Section 04 (CodeBlock) and Section 11 (Table) both hit these and were
fixed in-flight.
**Suggested fix**: Update `prompts/step-b-writing.md` examples + any
`references/*` that show `<CodeBlock>` / `<Table>` usage to match the actual
reacticle 0.2.6 prop signatures. Add a one-liner near the top of
`component-policy.md` saying "as of reacticle 0.2.6 these are the prop shapes;
re-verify when bumping the dependency".

### F12 · `density.sh` regex defects → false fail on Section 04

**Severity**: med
**Area**: `scripts/audit/density.sh`
**Discovered**: dogfood batch F
**Reproduction**:
- `density.sh` does not recognize multi-line self-closing `<CodeBlock ... />`
  blocks (no closing tag match in its awk parser); awk then counts all subsequent
  lines as code-block body. Section 04's two `<CodeBlock code={`...`} />` blocks
  spuriously got counted as 200+ lines of code, blowing past the cap. Verdict:
  `codeCharShare=0.965 > 0.15 → fail`. Manual audit of the same section
  confirms it passes Q9b (2 blocks ≤ 2 cap, 4 + 6 lines per block ≤ 8 cap).
- Separately: `density.sh` greps for `<P>` (capital) to count paragraphs. Our
  content uses lowercase `<p>` (per reacticle convention) → `paragraphCount=0`
  always → `blockToParagraphRatio` always 0 (vacuous pass on that one
  dimension, which masks the other dimension's false fail).
**Suggested fix**: Update awk to recognize JSX self-closing tags
(`<CodeBlock ... />` end-of-block marker) and case-insensitive
`<P>`/`<p>` (or, better, count rendered paragraphs via a smarter parser).
v0.1.0 worked around by manually auditing density and using `--skip-audits`
in delivery.sh.

### F13 · `claim-trace.sh` regex defects → vacuous PASS

**Severity**: med
**Area**: `scripts/audit/claim-trace.sh`
**Discovered**: dogfood batch F
**Reproduction**: `claim-trace.sh`'s `extract_prose` function greps for `<P>` /
`<Quote>` / `<Callout>` (all capital). Our content uses lowercase `<p>` (per
reacticle convention) → no prose lines extracted → `samples=[]` for every
section → `verdict=pass` (vacuous). Anti-hallucination defense degrades to
"post-hoc manual spot-check" rather than automatic.

**Same root cause as F12**: skill's audit scripts hard-code component names
they expect Step B SubAgents to use; when reality drifts (because F6 forced
SubAgents to use `<Aside>` / `<Detail>` / native `<p>` instead), the audit
scripts silently mismatch.

**Suggested fix**: Either (a) update `claim-trace.sh` and `density.sh` to use
the actual reacticle 0.2.6 components AND case-insensitive matching; or (b)
revisit the whole audit-script approach — these are AST-level audits being
done with regex. A tiny TypeScript AST parser (e.g. via `@babel/parser`) on
the .tsx files would be far more robust.

### F14 · `Article.tsx` doesn't auto-inject coverage / freshness footer

**Severity**: low
**Area**: `scripts/scaffold.sh` output + `article/Article.tsx` template +
`references/review-checklist.md` §6.8
**Discovered**: dogfood batch F
**Reproduction**: `references/review-checklist.md` §6.8 ("Coverage / Freshness
audit-嵌入 footer") describes Article.tsx reading `review/coverage.json` and
`review/freshness-summary.md` and rendering a footer line:
"Coverage: <pct>% · <assigned> assigned / <annexed> annexed / <missing.length>
missing". Scaffold doesn't wire this up; main agent in dogfood didn't add it
either (was not in `plan.md` as a required step). Verdicts can be found via
`review/*.json` files but not in the article itself. Not a honesty-rule
breach for v0.1.0 (both verdicts are PASS, and `analysis-snapshot.json`
captures them), but for size tier `>10k` projects where coverage might be
< 100%, this missing wire-up would let `Article.tsx` silently omit the
required "本报告为 ~70% 覆盖归档版本" disclosure.
**Suggested fix**: Either (a) scaffold ships an Article.tsx skeleton that
imports a `<DeliveryFooter />` component reading `review/coverage.json` and
`review/freshness-summary.md`; or (b) `delivery.sh` post-processes
`article/article.html` to inject the footer line. (a) is cleaner; (b) is more
portable (works even if Article.tsx is hand-edited).

### F15 · `html-to-pdf.sh` Windows path handling: `file://` URL not Windows-compatible

**Severity**: med
**Area**: `scripts/html-to-pdf.sh` line 157 + Windows / Git Bash interop
**Discovered**: dogfood batch G (PDF export step)
**Reproduction**: On Windows + Git Bash, running
`bash html-to-pdf.sh ./article.html ./article.pdf`
emits:
```
▸ 用浏览器：/c/Program Files/Google/Chrome/Application/chrome.exe
▸ 输入：article/article.html
▸ 输出：article/article.pdf
✗ 浏览器返回成功但输出文件不存在：article/article.pdf
```
Root cause: the script passes `"file://$TMP_HTML"` (e.g.
`file:///tmp/beautiful-codebase-pdf.XXX/article-print.html`) to Chrome. Windows
Chrome cannot resolve POSIX-style `/tmp/...` paths; it silently returns
success (exit 0) but doesn't write the output. Similarly, `--print-to-pdf=$OUTPUT`
uses a POSIX path Chrome can't resolve on Windows.

Manual workaround that worked: `cygpath -w "$TMP_HTML"` to get
`C:\Users\YANGZH~1\AppData\Local\Temp\...` and `cygpath -w "$OUTPUT"` for
the output, then pass `file:///$WIN_TMP_HTML` and `--print-to-pdf=$WIN_OUTPUT`.
PDF produced: 1.6 MB, cover visible on first page.

**Suggested fix**: Detect Windows / Git Bash environment (`uname -s` contains
`MINGW` / `CYGWIN` / `MSYS`) and translate paths via `cygpath -w` before
invoking Chrome. Or, more portably, copy the temp HTML to a path under the
output directory first (so input + output share the same drive) and use
relative paths. Without this fix, the PRD's AC11 (PDF export) cannot be
satisfied via `delivery.sh --pdf` on Windows; only manual fallback works.
